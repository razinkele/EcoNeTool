# EcoNeTool — Deep Analysis Report

_Generated 2026-07-17 by a 12-subsystem multi-agent workflow (253 agents; 80 raw findings → 55 survived 3-lens adversarial verification → 44 ranked after dedup). Two subsystem finders (external-apis, schema-sync) and several verify agents were lost to transient API-overload (529) errors, so those two areas are under-covered — see Coverage gaps._

## Architecture

EcoNeTool is a single-process bs4Dash Shiny app whose composition root is app.R (sourced from the 13-line run_app.R). At process start app.R sources config (R/config.R builds COLOR_SCHEME, METAWEB_PATHS, API_KEYS-as-environment, runs validate_metaweb_paths), then plugins/harmonization config, validation_utils.R (%||%, with_timeout, is_valid_value), the Phase-6 optional files (including parallel_lookup.R which sets a PROCESS-GLOBAL future plan), domain functions, the 17 UI files (one per tab), the ~22 server modules, and finally R/data_loading.R which leaves globals `net` (igraph) and `info` (data.frame) in globalenv. Per session, the flat `server()` mints the shared reactive spine — net_reactive/info_reactive (info_reactive derives `colfg` from fg names → get_functional_group_levels() → COLOR_SCHEME index), trophic_levels_cached, current_metaweb, ecopath_import_data, euseamap_data, plugin_states, and shared_trait_data — and threads it into ~14 plain-function modules that all register outputs on one shared, flat output namespace. Only rpathModuleServer uses moduleServer/ns(); everything else takes (input, output, session) directly, which is why the one duplicate id (mti_heatmap) is safe.

Data enters through three ingest fronts that all converge on the (net, info) pair. (1) EcoPath import (R/modules/ecopath_import_server.R) forks into a native Access path (ecopath/ecopath_windows.R RODBC on Windows, ecopath/ecopath_unix.R Hmisc/mdbtools on the Linux deploy target) and a CSV/Excel path (ecopath/ecopath_csv.R), builds a prey×predator diet matrix → igraph, assigns functional groups via functional_group_utils.R, and writes net/info + a metaweb. (2) Trait Research (R/modules/trait_research_server.R) takes a species list and fans each name through the trait pipeline. (3) Spatial (R/modules/spatial_server.R) tessellates a study area into a hex grid, enriches cells with EMODnet EUSeaMap habitat (emodnet_habitat_utils.R, euseamap_regional_config.R), and extracts per-cell local food webs from the metaweb.

The trait pipeline is the deepest stack. trait_lookup/orchestrator.R::lookup_species_traits() is a ~1590-line straight-line function: RDS-cache probe under cache/taxonomy/ (30-day TTL) → offline SQLite DB (cache/offline_traits.db via lookup_offline_traits) → WoRMS taxonomy → taxonomy-based smart routing over ~18 sources → per-source raw-trait accumulation → harmonization → ML/phylogenetic imputation → uncertainty quantification → RDS write. Layer 2a (trait_lookup/database_lookups.R + csv_trait_databases.R + api_trait_databases.R) does per-source retrieval into a uniform {success, traits, error} envelope; Layer 2b (trait_lookup/harmonization.R + R/config/harmonization_config.R) converts raw evidence into the categorical MS/FS/MB/EP/PR (+RS/TT/ST) vocabulary via ordered regex cascades. A parallel offline writer (scripts/initialization/build_offline_trait_db.R) reproduces this classification for the SQLite DB. batch_lookup_traits() loops the orchestrator and rbinds.

Outputs fan back out from net/info: visNetwork food webs (network_visualization.R, pairing V(net)$name[i] with info$fg[i]/colfg[i] positionally), flux + keystoneness (analysis_server.R, fluxweb), DT tables, leaflet maps, and downloads. The ECOPATH/ECOSIM analysis tab (R/modules/rpath_server.R + R/functions/rpath/*) is a self-contained branch converting an imported EwE model into an Rpath.params object, running mass balance, MTI, Ecosim, and a DATRAS/SAG survey-trend diagnostic. Persistence lives under the deploy tree (data/, cache/offline_traits.db, r-libs/, feedback.db, .Renviron) on laguna.ku.lt, reloaded by `touch restart.txt`; production shiny-server.conf has preserve_logs commented out — a fact that quietly shapes half the findings.

## Overall health

EcoNeTool is a large, feature-rich research app whose core plumbing (module wiring, timeout discipline, the PR9α session-isolation fixes, feedback_store.R, the survey-trends block) shows genuine, hard-won engineering maturity — but that maturity is unevenly distributed, and the un-hardened surfaces carry serious risk. The dominant hazard is silent scientific corruption: at least six confirmed defects (FS0 producer inversion, ecopath_csv biomass misalignment, the diet-matrix off-by-one, PTDB TL 1.0 default, the 1-degree habitat truncation, and the always-NA/always-erroring rpath diagnostics) produce plausible-looking wrong numbers that flow to exports and publications with a success toast, which is worse than a crash for a scientific tool. Compounding this, two deploy scripts can irreversibly destroy production state (.Renviron, feedback, r-libs) on their documented paths, and CI does not run the offline test suite on push/PR, so regressions in the trait pipeline and untested trait_lookup/ files reach the server ungated. The recurring root causes are structural and fixable — message()-not-warning() drift, positional vector coupling that should be name-keyed joins, unguarded zero-length/NA operands, and two-writer schema drift — and concentrating remediation on those four patterns plus the deploy/CI gaps would materially raise the whole codebase's floor.

## Cross-cutting themes

### Silent scientific corruption is preferred over loud failure
*Subsystems:* trait-harmonization, trait-db-lookups, spatial, ecopath-import, rpath-module

The highest-value pattern: multiple defects produce plausible-looking WRONG numbers that flow into exports and publications with no error. FS0's primary-producer regex matches diet nouns ('plant','algae','diatom') and is tested first, so herbivores/planktivores are entered into the food web as autotrophs — inverting trophic structure. PTDB's unknown Trophic_strategy silently defaults heterotrophs to primary_producer at TL 1.0. The spatial habitat step truncates coverage to a ~1x1 degree box and launders the missing 90% into 'No data'/diversity 0 grid cells. ecopath_csv's recycled logical mask attaches biomass/PB/QB to the wrong species. The diet-matrix cell edit writes to the next prey row. Each imports 'successfully' and looks like a data problem, not a code bug.

### message() where the house rule mandates warning() — the single most repeated defect
*Subsystems:* trait-orchestrator, trait-db-lookups, caching, rpath-module, trait-research-module, app-wiring, ecopath-import, spatial

Because production shiny-server.conf has preserve_logs commented out, message() output is destroyed on session end. Error handlers using message() instead of warning() appear in at least eight files: orchestrator.R (L1430/1372/1389), database_lookups.R (L51/54/161/164/1067 plus 4 bare NULL swallows), cache_sqlite.R (L785), rpath_server.R (L784/1100/1207), trait_research_server.R (L1107 + the offline_db_summary catch-all), config.R (L231), ecopath_windows.R (8 sites), spatial_server.R (~30 cat()/message()), plus PTDB's default branch. The convention is documented and followed correctly in newer code (feedback_store.R, survey_validation.R, the orchestrator's phylo tier), which makes the violations pure drift — the same class of bug PR2 already paid to find.

### Parallel vectors coupled by row index, with the contract enforced by nobody
*Subsystems:* app-wiring, ecopath-import, caching

net/info pairing is positional throughout: create_foodweb_visnetwork pairs V(net)$name[i] with info$fg[i]/colfg[i]/meanB[i] by index, and the only guard is a stop() on missing column names. Writers that violate 'info rows in vertex order, with colfg' silently draw the wrong species: data_import_server's parse_adjacency_df never reorders info to vertex order (and merge() re-sorts alphabetically); metaweb_manager export omits colfg entirely; dataeditor coerces numeric columns to character. The same positional fragility recurs in ecopath_csv (mask computed on shrunk vector, applied to full frame) and in the cache layer (classify_species_api NULL payload → list element deletion shifts every trait row against its species name). A shared finalize_network(net, info) helper would collapse the app-wiring instances.

### Zero-length / NA operands to && and if(), from unguarded partial upstream responses
*Subsystems:* trait-orchestrator, trait-db-lookups

R >= 4.3 makes a length-0 or length-n operand to && a hard error, and if(NA) always errors. The routing and lookup code repeatedly evaluates taxonomy/attribute values that can legitimately be NULL/NA/character(0): orchestrator crashes when WoRMS returns success=TRUE with NULL phylum (timeout) or NA isMarine (phytoplankton); database_lookups crashes on zero-length traits$class (discarding the ENTIRE WoRMS record), on AlgaeBase's data-frame-as-list mishandling, and on SHARK's length-n depth column. The correct idiom (isTRUE(), length checks, is_valid_value()) exists in the same files but is applied inconsistently — the phylo tier is hardened, the routing block one function away is not.

### Two-writer / two-reader schema drift on shared config and data
*Subsystems:* trait-harmonization, caching, ecopath-import, deploy-config

The same config or data store is consumed by parallel code paths that have diverged. harmonize_protection/harmonize_environmental_position hard-code inline regexes while build_offline_trait_db.R loops HARMONIZATION_CONFIG$*_patterns — so the same raw text yields different codes online vs offline, and the config comment asserts a correspondence that does not exist. cache/taxonomy/ is shared by two writers (orchestrator {traits,...} vs classify_species_api {data,...}) with colliding keys and incompatible envelopes. Windows vs Linux ecopath parsers return divergent metadata shapes; the consumer only reads Windows fields, so enrichment silently no-ops on the actual Linux deploy target. CI's hardcoded directory list drifted from the on-disk layout (trait_lookup/ never parse-checked).

### Dead or inert UI controls that actively mislead
*Subsystems:* trait-research-module, trait-harmonization, app-wiring, rpath-module

A recurring pattern of UI accreted ahead of, or drifted from, server wiring — and it lies to the user rather than failing visibly. The Databases checkbox group (9 options, 6 inert) is ignored while the progress modal reports looking up species in the deselected databases. The entire ecosystem-profile subsystem is inert (active_profile never written) while the UI affirmatively shows a '1.3x multiplier'. Harmonization threshold sliders revert every edit (self-referential observer reloads JSON). Rpath's Visualization Options, Update Plot button, and Auto-balance checkbox are all no-ops. Three trait-research buttons have no handler at all. Save Changes in rpath is a pure notification.

### Process-global state on a shared Shiny Server OSS process
*Subsystems:* app-wiring, spatial, caching

Shiny Server OSS runs one R process for all sessions — the premise PR9α relied on when fixing HARMONIZATION_CONFIG globalenv contamination. The same class survives in code that review didn't cover: the future plan is process-global and its only teardown hook is a cross-session kill switch; sf_use_s2(FALSE) is left globally disabled for the whole process if a BBT load throws (unlike the five other call sites that save/restore); cache directories are unnamespaced shared state. Session-scoping was applied surgically to the trait config but not generalized.

### The orchestrator's straight-line mutable-accumulator design is the substrate for its own bug class
*Subsystems:* trait-orchestrator

lookup_species_traits() is one ~1590-line function where all merge state lives in mutable locals and control flow (early exit, offline prefill, fallback) is ad-hoc guards rather than a table. Source priority is implicit in call order plus scattered is.null() guards, applied inconsistently: MS/FS/MB/EP/PR are first-writer-wins, RS/TT/ST are last-writer-wins with no NA guard, and the offline_prefilled guard covers 4 of 5 blocks (MS missed). The return column set is path-dependent, which is exactly why batch rbind fails. Nearly every high-severity trait finding traces back to this shape.

### Deploy scripts disagree about what persistent state must survive
*Subsystems:* deploy-config

Four deploy paths carry mutually inconsistent notions of persistent state. Only deploy-windows.ps1 partially encodes the invariant (data/, cache/, r-libs/ must survive), and it contradicts itself by printing a ready-to-paste `sudo rm -rf /srv/.../EcoNeTool/*` on its documented happy path. root deploy.sh's rsync --delete deletes .Renviron, r-libs/ and data/feedback/ (none excluded, none present locally). NO path protects .Renviron — the file carrying the very config meant to keep state safe. CI coverage compounds it: the offline test suite gates nothing on push/PR.

### Detail

#### #1 [critical] deploy.sh rsync --delete destroys production .Renviron, r-libs/ and data/feedback/ on every remote deploy
- **Location:** `deploy.sh:457`
- **Why it matters:** Runs on every ./deploy.sh; .Renviron (FEEDBACK_DB_PATH, FEEDBACK_ADMIN_KEY), r-libs/ (icesSAG) and all user feedback are deleted with no exclude and no local counterpart, while the script logs 'Deployment successful!'. Irreversible data loss disguised as success.
- **Fix:** Never rsync the repo root with --delete over the live tree; explicitly exclude .Renviron, r-libs/, data/, cache/, or deploy only an enumerated file set to a staging dir then cp -rT.

#### #2 [critical] deploy-windows.ps1 prints `sudo rm -rf /srv/shiny-server/EcoNeTool/*` on the documented -NoSudo path
- **Location:** `deploy-windows.ps1:693`
- **Why it matters:** Terminal output of the documented happy path is a copy-paste command that wipes data/ (feedback.db), cache/offline_traits.db and r-libs/; because -SkipData was used, the subsequent cp never restores data/. Most likely destructive defect to actually fire.
- **Fix:** Replace the printed NEXT STEPS with the safe `cp -rT staging /srv/.../EcoNeTool` workflow from project memory; never emit a blanket rm -rf against the live tree.

#### #3 [critical] batch_lookup_traits() rbind fails whenever two species take different return paths
- **Location:** `R/functions/trait_lookup/orchestrator.R:1722`
- **Why it matters:** Return column set is path-dependent (offline-complete/no-data return 36 cols; full path appends *_confidence/interval/overall_confidence/ml columns). do.call(rbind) errors on mismatched columns, and batch_lookup_traits has no tryCatch, so an entire multi-species run (the primary food-web-build use case, potentially hours of API calls) is discarded.
- **Fix:** Use dplyr::bind_rows() (as trait_research_server already does) instead of rbind, or normalize every return path to a fixed column superset before returning.

#### #4 [critical] cache/taxonomy .rds shared by two writers with colliding keys and incompatible payload schemas
- **Location:** `R/functions/taxonomic_api_utils.R:977`
- **Why it matters:** classify_species_api writes {data,timestamp}; the orchestrator/trait_research expect {traits,...}. For a plain binomial both keys collapse to the same filename and both carry $timestamp so the TTL guard passes. Import-then-trait-research corrupts in both directions: NULL trait rows delete/shift list elements (species misaligned against traits) or the import aborts on is.na(NULL$source).
- **Fix:** Namespace the two caches into separate dirs, or add an envelope 'type'/version tag and validate shape on read before dereferencing $traits/$data.

#### #5 [critical] Diet-matrix cell edit writes to the wrong prey row (off-by-one)
- **Location:** `R/modules/rpath_server.R:968`
- **Why it matters:** DT _cell_edit returns 1-based row, 0-based col (rownames=FALSE); the diet handler adds 1 to both, so every diet edit lands on the next prey row and Ecopath balance runs on a matrix the user never specified. Editing the last row grows one data.table column to nrow+1, producing an invalid data.table that errors on render/rpath().
- **Fix:** Use row <- info$row unchanged and col_name <- cols[col + 1], matching the group-parameters handler in the same file (L848/854).

#### #6 [critical] ecopath_csv logical mask from a shrunk species vector is applied to the full basic_data, misaligning biomass/PB/QB
- **Location:** `R/functions/ecopath/ecopath_csv.R:73`
- **Why it matters:** valid_rows is computed from the already-NA-dropped species_names but subsets the full-length basic_data; R silently recycles the short mask, so biomass/PB/QB are read from mis-subset rows and zipped positionally to species. A single blank group name (routine in hand-edited exports) attaches wrong parameters to every downstream trophic-level/keystoneness metric. Imports as SUCCESS.
- **Fix:** Compute valid_rows against the original full-length frame before dropping, or drop rows once and derive all vectors from the same filtered frame; assert length equality.

#### #7 [critical] FS0 primary-producer regex matches diet nouns and is tested first, coding consumers as autotrophs
- **Location:** `R/config/harmonization_config.R:21`
- **Why it matters:** FS0 includes 'plant|algae|phytoplankton|diatom'; harmonize_foraging_strategy tests FS0 before FS4/FS6 and paste()-collapses all sources, so any source contributing 'algae'/'plant' (e.g. FishBase feeding_type 'grazing on aquatic plants', TraitBank diet 'diatoms') forces FS0 for the whole species. FS0 is the base of the food web — this silently inverts trophic structure in trait_foodweb.R interaction probabilities.
- **Fix:** Remove prey/diet nouns from FS0; require producer-specific terms (photosyn|autotroph|producer); reorder so FS4/FS6 win over ambiguous diet text, and match modality fields not raw diet prose.

#### #8 [critical] WoRMS classification timeout returns success=TRUE with NULL phylum, crashing the routing block
- **Location:** `R/functions/trait_lookup/orchestrator.R:355`
- **Why it matters:** success is length(traits)>0 and isMarine/aphia_id are set unconditionally, so a classification timeout (the routine case the timeout exists for) yields success=TRUE without phylum. tolower(NULL) → character(0), and `phylum == 'chordata' && ...` inside if() errors on R>=4.3. No tryCatch around routing, so the whole lookup aborts and (under batch) kills every remaining species instead of entering the intended fallback.
- **Fix:** Guard routing with is.null/length checks or route to the WoRMS-FAILED fallback whenever phylum is NULL/empty; wrap the routing block in tryCatch.

#### #9 [critical] Habitat coverage silently capped to a ~1x1 degree box; the Clip button cannot restore it
- **Location:** `R/modules/spatial_server.R:831`
- **Why it matters:** Enabling habitat loads only a +/-0.5 degree box around the study-area centre (hard wkt_filter on the GDB path). Any study area wider than ~1 degree is truncated at load; clip_habitat_to_study_area only intersects the truncated cache and never re-reads the GDB. overlay stamps 'No data'/diversity 0 on outside cells, so 90% missing seabed is exported as a normal result with a success toast.
- **Fix:** Pass the study-area bbox (not a fixed centre box) to load_regional_euseamap, or have the clip step re-read the GDB with the study-area extent; warn when the loaded extent is smaller than the study area.

#### #10 [high] isMarine = NA from WoRMS crashes the phytoplankton/algae routing branch
- **Location:** `R/functions/trait_lookup/orchestrator.R:424`
- **Why it matters:** WoRMS returns NA isMarine for most phytoplankton/algae; the guard defends only NULL, so `!is.null(is_marine) && is_marine==FALSE` → TRUE && NA → NA and if(NA) errors, aborting the lookup with no fallback and killing the rest of a batch. The isTRUE() idiom used elsewhere in the same file for exactly this hazard is not applied.
- **Fix:** Use isTRUE(is_marine==FALSE) / isFALSE(...) or coerce NA to a routing default; wrap routing in tryCatch.

#### #11 [medium] EMODnet habitat enrichment calls undefined current_network(), feature 100% broken and disguised as a missing file
- **Location:** `R/modules/ecopath_import_server.R:1600`
- **Why it matters:** current_network is not a parameter and defined nowhere (intended current_metaweb). The tryCatch reports 'Failed to load EUSeaMap... ensure EUSeaMap_2025.gdb exists', so users hunt for a present file while the feature can never enable on any input.
- **Fix:** Rename the three calls to current_metaweb; add a test that ticks the checkbox.

#### #12 [high] Trophic pyramid plot always errors: reads $TL from params object which has no TL column
- **Location:** `R/modules/rpath_server.R:1682`
- **Why it matters:** params$model (create.rpath.params output) has no TL; only the balanced ecopath_model does. living$TL is NULL → max(NULL) = -Inf → seq(1, ceiling(-Inf)) throws, unconditionally, for every dataset. Secondary: TL=1.0 producer/detritus groups fall outside all bins once fixed.
- **Fix:** Read rpath_values$ecopath_model (has $TL and lowercase $type); use breaks starting below 1.0 so TL=1 groups are binned.

#### #13 [high] Diagnostics reports 'Mean Trophic Level: NA' always — same params/balanced schema confusion
- **Location:** `R/modules/rpath_server.R:1628`
- **Why it matters:** Guards on ecopath_model but reads params$model$TL (NULL) → mean(NULL)=NA, silently, with 'Diagnostics calculated!' shown. A headline ecosystem metric renders NA for every model, looking like a user data problem.
- **Fix:** Read TL and Type from rpath_values$ecopath_model, not params$model.

#### #14 [high] Offline-prefilled MS silently wiped to NA while MS_source stays 'OfflineDB'
- **Location:** `R/functions/trait_lookup/orchestrator.R:1080`
- **Why it matters:** The MS harmonization else-branch lacks the !'MS' %in% offline_prefilled guard the other four traits have, so when the offline DB supplies MS but no API returns a size, the known MS is overwritten with NA (source still claims OfflineDB) and the ML tier wastefully re-imputes a trait already held.

## Subsystem maps

### trait-orchestrator
**Purpose.** Orchestrates a single species' trait request across ~18 trait sources (offline SQLite DB, WoRMS taxonomy, an ontology fuzzy-trait table, FishBase/SeaLifeBase/BIOTIC/BVOL/SpeciesEnriched/freshwater/MAREDAT/PTDB/AlgaeBase, five bundled CSV DBs, and five APIs), harmonizes raw values into the categorical MS/FS/MB/EP/PR (+RS/TT/ST) codes, applies ML and phylogenetic imputation for gaps, quantifies confidence, and caches the result. `batch_lookup_traits()` fans this over a species list and rbinds.

**Entry points:** lookup_species_traits(species_name, biotic_file, maredat_file, ptdb_file, cache_dir) -- L111; batch_lookup_traits(species_list, ...) -- L1708; lookup_offline_traits(species_name, db_path = 'cache/offline_traits.db') -- L46; load_all.R sources the 5 module files in dependency order (database_lookups -> csv -> api -> harmonization -> orchestrator)

**Key files:** `R/functions/trait_lookup/orchestrator.R`, `R/functions/trait_lookup/load_all.R`, `R/functions/trait_lookup/database_lookups.R`, `R/functions/trait_lookup/harmonization.R`, `R/functions/trait_lookup/csv_trait_databases.R`, `R/functions/trait_lookup/api_trait_databases.R`, `R/functions/uncertainty_quantification.R`, `R/functions/ml_trait_prediction.R`, `R/functions/phylogenetic_imputation.R`

**Data flow.** lookup_species_traits(species_name) -> (1) RDS cache probe in cache_dir (30-day TTL, key = gsub(' ','_',species)); (2) a 36-column `result` data.frame of NA placeholders is initialised (L141-161); (3) WoRMS taxonomy [1/12] populates raw_traits$worms (phylum/class/isMarine/aphia_id/max_length_cm/habitat); (4) ontology fuzzy traits [2/12]; (5) offline SQLite DB (`cache/offline_traits.db`) via lookup_offline_traits() -- if MS+FS+MB+EP+PR are all non-NA it EARLY-RETURNS with confidence hardcoded 'high' (L283-307), else pre-fills the non-NA subset and records names in `offline_prefilled`; (6) SMART ROUTING sets ~18 query_* booleans purely from tolower(phylum)/tolower(class) (L347-468); WoRMS failure flips to fallback 'query everything' mode; (7) each enabled source appends into local accumulators `size_cm` / `trophic_level` / `feeding_mode` / `mobility_info` / `habitat_info` / `protection_info` / `body_shape` / `depth_min|max` (first-writer-wins for scalars via `is.null(x) &&` guards; character vectors just concatenate); check_completeness() can EARLY-EXIT the [4/12]-[11/12] block; (8) CSV+API sources additionally write RS/TT/ST directly onto `result` with no priority guard; (9) HARMONIZATION converts each accumulator to a code via harmonize_* (falling back to harmonize_fuzzy_* over raw_traits$ontology when an accumulator is empty), guarded by `!'X' %in% offline_prefilled`; (10) ML fallback (needs raw_traits$worms) fills NA codes, sets *_source='ML', imputation_method='rf_predicted'; (11) uncertainty_quantification's calculate_all_trait_confidence() merges *_confidence / *_confidence_category / interval columns onto `result`, overall = geometric mean; (12) phylogenetic imputation (needs raw_traits$worms AND cache_dir); (13) `result$source` = unique(sources_used) joined by '+', categorical `confidence` from overall_confidence else count-of-5; (14) saveRDS of list(traits, harmonized, species, timestamp). batch_lookup_traits() loops and `do.call(rbind, results_list)`.

**Dependencies:** validation_utils.R (%||%, with_timeout, is_valid_value) - loaded by app.R, NOT sourced here; R/config/harmonization_config.R (HARMONIZATION_CONFIG) - existence check at L28-35 self-sources as fallback; DBI + RSQLite (offline_traits.db); worrms / rfishbase / httr / jsonlite (via database_lookups.R + api_trait_databases.R); R/functions/local_trait_databases.R - lazily sourced inside the reactive path at L588 and L634; ml_trait_prediction.R::apply_ml_fallback, phylogenetic_imputation.R::apply_phylogenetic_imputation, uncertainty_quantification.R::calculate_all_trait_confidence - all resolved by exists() at call time

**Observations.** The orchestrator is a single ~1590-line straight-line function; all merge state lives in mutable locals, so control flow (early exit, offline prefill, fallback mode) is expressed by ad-hoc guards rather than a table. Source priority is implicit in call ORDER plus scattered `is.null(size_cm) &&` guards, and is applied inconsistently: MS/FS/MB/EP/PR scalars are first-writer-wins, but RS/TT/ST are last-writer-wins with no NA guard. The `offline_prefilled` guard is applied in 4 of 5 harmonization blocks (MS is missed). Recent hardening is visible and good: the defensive dbListFields SELECT (L86-87), isTRUE() around phylo source checks (L1542-1546, with the NA rationale in comment), and warning()-based surfacing of skipped ML/phylo tiers (L1279, L1296, L1509, L1521). That rule is not yet applied to the three error handlers at L1372/L1389/L1430, which still use message(). The return value's column set is path-dependent, which the map's final rbind does not tolerate. Routing is string-equality on taxonomy with no is.null/NA defence, so any WoRMS partial success is a hard crash rather than a degraded lookup. Two lazy `source()` calls remain inside the request path (L588, L634) -- the same race the ML/phylo tiers were explicitly hardened against.

### trait-db-lookups
**Purpose.** Per-source trait retrieval layer for the trait pipeline. Provides 9 exported `lookup_*_traits()` functions that each query one external trait source (FishBase, SeaLifeBase, AlgaeBase, SHARK, freshwaterecology.info, BIOTIC, MAREDAT, PTDB, WoRMS) plus a local ontology CSV, normalise the response into a common result envelope, and hand raw (un-harmonised) traits to the orchestrator. This file is Layer 2a: it does no harmonisation and no MS/FS/MB/EP/PR coding.

**Key files:** `R/functions/trait_lookup/database_lookups.R (the subsystem itself, 1178 lines, 9 lookup fns + 2 internal helpers `get_ptdb_data`, `worms_records_by_name_http`)`, `R/functions/trait_lookup/load_all.R (source order: database_lookups -> csv_trait_databases -> api_trait_databases -> harmonization -> orchestrator; explicit sourcing, so the stray `api_trait_databases-laguna-safeBackup-0001.R` in the same dir is NOT loaded and causes no collision)`, `R/functions/trait_lookup/orchestrator.R (sole consumer; `lookup_species_traits()` routes taxonomically and merges `$traits` into `raw_traits$<source>`)`, `R/functions/validation_utils.R (`with_timeout`, `safe_get`, `is_valid_value`, `coalesce_values` — the NULL/NA-safety contract this file depends on)`, `R/config.R (`get_api_key()` — returns "" not NULL when unset, so the line-384 `api_key == ""` guard is safe)`, `data/ontology_traits.csv (2704 rows, 11 cols, fuzzy-scored ontology traits)`, `R/functions/taxonomic_api_utils.R (independent second FishBase reader; its `Weight` handling contradicts this file's — see findings)`

**Data flow.** orchestrator.R `lookup_species_traits(species_name)` -> taxonomic routing flags (query_algaebase / query_ptdb / ...) derived from the WoRMS phylum+class fetched in step [1/12] -> each gated `lookup_<src>_traits(species_name)` call -> inside each fn a uniform envelope `list(species=, source=, success=FALSE, traits=list())` is pre-built, then a `tryCatch({...}, error=function(e) result$error <<- conditionMessage(e))` populates it -> `result$success <- length(traits) > 0` -> orchestrator gates on `isTRUE(x$success)` and copies `x$traits` into `raw_traits$<src>` -> harmonization.R converts raw traits to MS/FS/MB/EP/PR codes. Three source classes: (a) remote R packages via `with_timeout()` (rfishbase for FishBase/SeaLifeBase, worrms for WoRMS classification/attributes); (b) direct HTTP via `httr::GET` + `httr::timeout()` (WoRMS AphiaRecordsByName, freshwaterecology.info) — chosen deliberately because `setTimeLimit()` cannot interrupt a blocking curl read; (c) local CSV via `read.csv` (BIOTIC, MAREDAT, PTDB, ontology), of which only PTDB is memoised (`.ptdb_cache` env, keyed by `basename()`).

**Dependencies:** rfishbase (FishBase + SeaLifeBase, duckdbfs/parquet backend); worrms (wm_classification, wm_attr_data; wm_records_name only in the AlgaeBase path); httr + jsonlite (direct REST for WoRMS AphiaRecordsByName and freshwaterecology.info); shark4r (SHARK; optional, and the call site is commented out in the orchestrator); utils (read.csv, URLencode); internal: validation_utils.R helpers; R/config.R get_api_key()

**Observations.** Consistent, well-observed envelope contract (`success`/`traits`/`error`), and the house `<<-`-in-error-closure rule is followed correctly in all 9 error handlers — the `<-` vs `<<-` hunt turned up nothing; note `timeout_count <- timeout_count + 1` at line 899 is correctly plain `<-` because a `tryCatch` expr evaluates in the calling frame, so it is not a bug. Timeout handling is the most mature part of the file: the comments at 40-45, 773-780 and 880-883 document a real, correct distinction between setTimeLimit-interruptible R-level code and curl-level blocking reads. ### trait-harmonization
**Purpose.** Converts heterogeneous raw trait evidence (free text from FishBase/BIOTIC/WoRMS/TraitBank/regional CSVs, numeric sizes, and fuzzy-scored ontology rows) into EcoNeTool's categorical trait vocabulary: MS1-MS7 (size), FS0-FS7 (foraging), MB1-MB5 (mobility), EP1-EP4 (environmental position), PR0-PR8 (protection), plus RS/TT/ST extended modalities. harmonization_config.R is the declarative rule store (regex vocabularies, size thresholds, taxonomic inference toggles, ecosystem profiles, human-readable labels); harmonization.R is the rule engine.

**Data flow.** Two independent paths converge on the same trait codes. (1) LIVE: orchestrator.R queries external sources, concatenates each source's free text into shared character vectors (feeding_mode, habitat_info, mobility_info, protection_info), then calls harmonize_*(); each helper does tolower(paste(x, collapse=' ')) and runs an ordered first-match-wins regex cascade, falling back to taxonomic inference (phylum/class gated by is_rule_enabled()) and then a hard-coded default. **Dependencies:** shiny (getDefaultReactiveDomain for session-scoped config; soft dependency via requireNamespace); jsonlite (save/load_harmonization_config); base R regex (grepl) — the entire classification engine; %||% infix operator (defined elsewhere in the codebase); orchestrator.R (caller), build_offline_trait_db.R (parallel consumer of the same config)

**Observations.** The PR9-alpha session-isolation fix is architecturally sound and consistently applied inside harmonization.R — every live read routes through get_harm_config(). The real structural weakness is that the config is only *half* authoritative: harmonize_mobility and harmonize_foraging_strategy read patterns from config, but harmonize_protection and harmonize_environmental_position hard-code their regexes inline while the offline build script reads the config versions of those same vocabularies — a genuine two-writer/two-reader split that the config's own comment ('matching harmonize_protection') asserts does not exist. Secondary theme: the whole ecosystem-profile subsystem (profiles, size_multiplier, size_thresholds_adjust, apply_size_adjustment) is inert — no code path ever writes active_profile, so it is permanently 'temperate'. The regex cascades are order-dependent and untyped, which makes them fragile: substring collisions (subtidal/tidal) and diet-vs-modality conflation (plant/algae in FS0) both stem from matching prose with unanchored alternations. The extended RS/TT/ST helpers derive their output code from the loop *index* rather than the pattern name, which is safe for the built-in config but unsafe across the JSON import path. Strengths: confidence helpers are properly centralised, coral_thermal_to_tt is a good example of collapsing duplicated banding into one NA-safe helper, and the *_labels lists genuinely prevent UI/legend drift.

**Observations.** WMS: there are NO WMS layers in this subsystem or anywhere in R/ -- `grep -rn 'addWMSTiles' R/` returns nothing. CRS discipline is generally sound and the subsystem is careful in ways that suggest hard-won lessons: create_hexagonal_grid() correctly builds hexes in a projected UTM CRS rather than in degrees (so cell_size really is metres); every CRS check follows the same defensive `epsg NULL/NA -> assign 4326, else transform` shape; coordinate order is consistently (xmin, ymin, xmax, ymax) and I found no lon/lat transposition. The s2 toggling is the notable exception -- 4 of 6 call sites use the correct `old <- sf_use_s2(); on.exit(sf_use_s2(old))` save/restore idiom, and the two that don't are both in spatial_server.R (see findings).

The dominant design smell is a *state-machine that isn't one*. The habitat pipeline is a 3-step manual sequence (enable -> clip -> overlay) with each step gated only by req() on the previous step's output, and no step invalidates downstream state. Consequences: spatial_hex_grid() is destructively overwritten by the overlay result (line 1031), making the overlay non-idempotent; clipping is a pure re-clip of whatever bbox-truncated data the enable step happened to cache, so the enable step silently caps scientific coverage that no later step can restore; and changing the study area after enabling habitat leaves the stale euseamap_data() cache in place (spatial_clear_study_area clears the clip and grid-with-habitat but never euseamap_data(), which is shared with ecopath_import_server).

A second theme is tolerant-reader / strict-writer schema drift against EUSeaMap. get_habitat_at_location() goes to great lengths to accept ~10 alternative field names per attribute (documented: 'Field names may vary by EUSeaMap version'), while overlay_habitat_with_grid() and the leaflet popups hardcode the EUSeaMap_2025 names (EUNIScomb, EUNIScombD, Substrate, Biozone, Energy). The two halves disagree about how stable the schema is.

**Data flow.** STARTUP (once per R process): app.R sources R/config.R (defines constants, runs validate_metaweb_paths(), builds API_KEYS as an env, loads VERSION) -> plugins/harmonization config -> validation_utils (%||%, with_timeout) -> Phase-6 optional files, incl. init_parallel_lookup(workers=4) which sets a PROCESS-GLOBAL future plan -> domain functions -> UI files -> module server files -> R/data_loading.R, which leaves globals `net` (igraph) and `info` (data.frame) in globalenv.

PER SESSION: server() repairs V(net)$name from info$species (local copy; globals untouched), then mints the shared reactive graph: net_reactive/info_reactive (info_reactive derives `colfg` by matching info$fg names against get_functional_group_levels() -> COLOR_SCHEME index), trophic_levels_cached / topological_metrics_cached (plain reactives; bindCache deliberately removed), current_metaweb, metaweb_metadata, dashboard_trigger, euseamap_data, ecopath_import_data, ecopath_native_status_data, diet_matrix_reactive, plugin_states, and shared_trait_data (reactiveValues bridging trait_research_server -> foodweb_construction_server).

Init order is load-bearing exactly once: dataeditor_inline_server runs before ecopath_import_server / data_import_server / ecobase_server because it RETURNS `refresh_data_editor`, a closure those three call after mutating net_reactive/info_reactive.

WRITE FAN-IN: net_reactive/info_reactive are written by data_import_server (upload), ecopath_import_server, ecobase_server, metaweb_manager_server (export-to-network) and dataeditor_inline_server (save). Convention on each write path: recompute `colfg`, call refresh_data_editor(), bump dashboard_trigger. READ FAN-OUT: dashboard_server (value boxes), visualization_server, analysis_server (flux + keystoneness, both bindCache'd on net+info), download_server, spatial_server. app.R tail pins the 5 dashboard boxes with outputOptions(suspendWhenHidden=FALSE); each module pins its own outputs the same way.

**Dependencies:** shiny, bs4Dash, shinyBS (bsModal for feedback/settings modals), shinyWidgets (switchInput in plugin_server); igraph (core network type carried by net_reactive), visNetwork, DT, leaflet, MASS (MTI pseudo-inverse), fluxweb; future + future.apply (parallel_lookup.R sets a process-global `plan`; hard-stops at source time if absent); jsonlite (config/api_keys.json, config/harmonization_custom.json); optional at call time: readxl, openxlsx, RCurl/XML/plyr/dplyr (EcoBase), Rphylopars, icesSAG via app-local r-libs/; process-global mutable state: globals `net`/`info` (data_loading.R), API_KEYS environment (deliberately an env so plugin_server can mutate in place), HARMONIZATION_CONFIG (read only via get_harm_config()), the future plan; session-global state: session$userData$harm_config, written by harmonization_settings_server

WEAKNESS 3 — self-referential observers. Several modules use `observe({ rv$x <- f(rv$x) })`, which both reads and writes the same reactiveValues slot with no isolate() (harmonization_settings_server 18-29/34-46; foodweb_construction_server 196-201). Shiny's identical() check on reactiveValues stops these from spinning forever, but in harmonization the two observers fight over rv$config and the JSON file wins (finding 2).

WEAKNESS 4 — errors reported via cat()/renderPrint into UI panes rather than warning(). config.R:231 is the one that clearly inverts the house rule; the cat() calls in dataeditor_inline_server refresh (44-46) are the same shape but land in a visible status box, so lower value.

TECH DEBT: metaweb_manager_server defines its own metaweb path list (30-39) with different keys from the global METAWEB_PATHS in config.R:143-149, so config.R's startup validate_metaweb_paths() validates paths the Metaweb Manager never reads. download_metaweb's CSV branch does setwd(tempdir()) in the shared process (guarded by on.exit, but visible to concurrent sessions). run_app.R's banner still says "Version 2.0" while VERSION/app.R say 1.4.4.
