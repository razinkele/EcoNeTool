# Automatic Versioning & Documentation System — Design Spec

**Date:** 2026-04-10
**Version target:** 1.4.3
**Status:** Approved for implementation

---

## 1. Overview

A comprehensive, pure-R versioning and documentation system for EcoNeTool that:

- Automatically determines version bumps from conventional commit messages
- Regenerates the CHANGELOG from the full git history
- Auto-generates an API reference from Roxygen2 source comments
- Updates all version references across the codebase atomically
- Creates annotated git tags
- Integrates with GitHub Actions for CI-driven releases and PR changelog previews
- Modernizes the README with current GitHub markdown features
- Provides a hand-written user manual for marine scientists

All release scripts are **pure base R** with zero external package dependencies.

## 2. Automatic Versioning Engine

### 2.1 Version Bump Script

**File:** `scripts/version_bump.R`

**Inputs:**
- Reads conventional commits since the latest `v*` git tag (or all commits if no tags exist)
- Accepts CLI override: `--bump patch|minor|major` or `--version X.Y.Z`

**Commit-to-bump mapping:**
| Commit prefix | Bump |
|---------------|------|
| `BREAKING CHANGE` in body/footer, or `!:` suffix | major |
| `feat:`, `feat(scope):` | minor |
| `fix:`, `perf:`, `docs:`, `test:`, `refactor:`, `chore:`, `ci:`, `build:`, `security:`, `style:` | patch |
| Non-conventional (e.g., "Add hover tooltips...") | patch |

The highest bump wins (major > minor > patch). Default when no commits found: patch.

**Files updated:**
| File | What changes | Method |
|------|-------------|--------|
| `VERSION` | All fields: VERSION, MAJOR, MINOR, PATCH, RELEASE_DATE, GIT_COMMIT, BUILD_DATE | Key=value rewrite |
| `app.R` | `# CURRENT VERSION: v1.4.2 (...)` line | Regex pattern match on `# CURRENT VERSION:` |
| `README.md` | Citation BibTeX version, version footer, badge URLs | Regex pattern replacements |

**Not updated:** `R/config.R` — it reads the VERSION file dynamically at runtime via `load_version_info()`, so it stays in sync automatically.

**VERSION_NAME:** Set via `--name "Release Name"`. If omitted, auto-generated from the most prominent `feat:` commit scope (e.g., "Trait Provenance & Performance"). Falls back to "Maintenance Release" if no feat commits.

### 2.2 Release Orchestrator

**File:** `scripts/release.R`

**CLI interface:**
```
Rscript scripts/release.R [--bump patch|minor|major] [--version 1.4.3] [--name "Name"] [--dry-run]
```

**Execution steps (in order):**
1. Validate clean working tree (no uncommitted changes)
2. Calculate new version (from commits or CLI override)
3. Run `version_bump.R` to update all version references
4. Run `generate_changelog.R` to regenerate CHANGELOG.md
5. Run `generate_api_reference.R` to regenerate API_REFERENCE.md
6. Stage all modified files
7. Commit with message: `chore(release): v{VERSION}`
8. Create annotated git tag `v{VERSION}` with changelog entry as message
9. Print summary and `git push --follow-tags` command (does NOT auto-push)

**`--dry-run` behavior:** Prints what would change (new version number, files affected, changelog preview, commit message) without modifying any files, committing, or tagging.

## 3. CHANGELOG Generation

### 3.1 Generator Script

**File:** `scripts/generate_changelog.R`

**Output:** `CHANGELOG.md` — full regeneration from git history on every release.

**Data source:** Parses all 119+ commits from `git log --format` with hash, date, subject, body.

### 3.2 Commit Parsing

**Conventional commit regex:**
```
^(feat|fix|perf|refactor|test|docs|chore|ci|build|security|style)(\(.+?\))?(!)?:\s*(.+)$
```

**Non-conventional fallback (9 early commits):**
Smart prefix guessing for commits that don't follow the convention:
- Starts with "Add" / "Reorganize" / "Initial commit" → Added
- Starts with "Fix" → Fixed
- Starts with "Clean up" / "Update" → Changed
- Everything else → Other

### 3.3 Output Format

```markdown
# Changelog

All notable changes to EcoNeTool will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

> Auto-generated by `scripts/generate_changelog.R` from conventional commits.

## [1.4.3] - 2026-04-10

### Added
- **traits:** add FishBase trait pre-fetcher for offline knowledge base (4e6aef1)

### Performance
- **traits:** 4 speedups for trait lookup (367db16)
- **traits:** remove redundant individual DB calls from trait research (692d4d5)

### Fixed
- **ui:** use bs4Dash status names for valueBox colors (44b1894)
- **review:** address 6 remaining issues (69e7244)

### Testing
- **traits:** add multi-regional species tests (0bd3451)
- **traits:** comprehensive trait lookup tests (f45f71e)

...

## [Unreleased]
(commits after latest tag, if any)

---

[1.4.3]: https://github.com/razinkele/EcoNeTool/compare/v1.4.2...v1.4.3
[1.4.2]: https://github.com/razinkele/EcoNeTool/releases/tag/v1.4.2
```

### 3.4 Version Grouping Logic

1. Find all `v*` annotated tags and their associated commits
2. Group commits between consecutive tags into version sections
3. Commits before the first tag → grouped into the earliest version section
4. Commits after the latest tag → "Unreleased" section at top
5. Within each version, sort by category (Added > Performance > Changed > Fixed > Security > Testing > Documentation > Maintenance > Other)

### 3.5 Category Mapping

| Commit prefix | Changelog section |
|---------------|-------------------|
| `feat:` | Added |
| `fix:` | Fixed |
| `perf:` | Performance |
| `refactor:` | Changed |
| `test:` | Testing |
| `docs:` | Documentation |
| `chore:`, `ci:`, `build:` | Maintenance |
| `security:` | Security |
| `style:` | Style |
| Non-conventional | Other |

**Breaking changes:** Commits with `!:` suffix or `BREAKING CHANGE` in body get an additional `### BREAKING CHANGES` section at the top of their version block.

### 3.6 Comparison Links

Repo URL extracted from `git remote get-url origin` at runtime, parsed to `https://github.com/{owner}/{repo}`. Compare links generated for each consecutive version pair.

## 4. GitHub Actions Workflows

### 4.1 Release Workflow

**File:** `.github/workflows/release.yml`

**Trigger:** `workflow_dispatch` with inputs:
- `bump_type` (choice: patch / minor / major)
- `version_override` (string, optional: explicit version like "2.0.0")
- `release_name` (string, optional: VERSION_NAME)

**Steps:**
1. Checkout with full history (`fetch-depth: 0` for tag access)
2. Setup R 4.4.1
3. Run `Rscript scripts/release.R --bump ${{ inputs.bump_type }}`
4. Push commit and tag
5. Create GitHub Release using `gh release create` with changelog entry as body

**Permissions:**
```yaml
permissions:
  contents: write
```

### 4.2 CI Enhancement

**File:** `.github/workflows/ci.yml` — add one job:

**Version drift detection (on push to master):**
- Check that VERSION file version matches latest git tag (if tags exist)
- Warning (not failure) if they've drifted — signals an unreleased version

### 4.3 PR Changelog Preview

**File:** `.github/workflows/auto-changelog.yml`

**Trigger:** `pull_request` to master

**Steps:**
1. Checkout with full history
2. Setup R 4.4.1
3. Run `Rscript scripts/generate_changelog.R --preview` (generates preview of what the PR's commits would add)
4. Post/update PR comment with preview using `gh pr comment`

**Permissions:**
```yaml
permissions:
  contents: read
  pull-requests: write
```

## 7. API Reference (Auto-generated)

### 7.1 Generator Script

**File:** `scripts/generate_api_reference.R`

**Output:** `docs/API_REFERENCE.md`

**Approach:** Custom Roxygen2 comment parser in pure base R. Does NOT use the `roxygen2` package (which requires a DESCRIPTION file / R package structure). Instead, parses `#'` comment blocks directly from source files using regex.

### 7.2 Parsing Rules

For each `.R` file in `R/`, `R/functions/` (including subdirectories `ecopath/`, `rpath/`, `trait_lookup/`), `R/modules/`, `R/ui/`, `R/config/`:
1. Find `#' @export` tagged functions
2. Extract Roxygen2 tags: `@title`, `@description`, `@param`, `@return`, `@examples`, `@family`, `@seealso`
3. Extract function signature from the line after the comment block
4. Record source file path and line number

### 7.3 Output Organization

Functions grouped by domain (not by file):

| Domain | Source directories/files |
|--------|------------------------|
| Network Analysis | `topological_metrics.R`, `trophic_levels.R`, `keystoneness.R`, `flux_calculations.R`, `network_visualization.R` |
| Data Import/Export | `ecopath/`, `rpath/`, `metaweb_core.R`, `metaweb_io.R`, `ecobase_connection.R`, `R/data_loading.R` |
| Trait Lookup | `trait_lookup/`, `local_trait_databases.R`, `trait_foodweb.R`, `trait_help_content.R`, `parallel_lookup.R` |
| Trait Imputation | `ml_trait_prediction.R`, `bhpmf_imputation.R`, `rphylopars_imputation.R`, `phylogenetic_imputation.R` |
| Spatial Analysis | `spatial_analysis.R`, `emodnet_habitat_utils.R`, `euseamap_regional_config.R` |
| Taxonomic APIs | `taxonomic_api_utils.R`, `shark_api_utils.R` |
| Shiny Modules | `R/modules/`, `R/ui/` |
| Configuration | `R/config.R`, `R/config/` |
| Utilities | `validation_utils.R`, `error_logging.R`, `cache_sqlite.R`, `api_rate_limiter.R`, `logger.R`, `auxillary_parser.R`, `functional_group_utils.R`, `uncertainty_quantification.R` |

All paths above are relative to `R/functions/` unless otherwise noted.

**Domain assignment:** Based on file path matching rules in the script. A mapping table in the script associates file path patterns to domain names.

### 7.4 Per-Function Entry Format

```markdown
### function_name()

**Description:** First line of Roxygen2 description

**Parameters:**

| Name | Type | Default | Description |
|------|------|---------|-------------|
| species_name | character | — | Scientific name to look up |
| timeout | numeric | 30 | API timeout in seconds |

**Returns:** Description from @return tag

**Source:** [`R/functions/trait_lookup/orchestrator.R:145`](../R/functions/trait_lookup/orchestrator.R#L145)
```

### 7.5 Header

```markdown
# EcoNeTool API Reference

> Auto-generated from source code — do not edit manually.
> Generated: 2026-04-10 | Version: 1.4.3
> Regenerate: `Rscript scripts/generate_api_reference.R`
```

## 10. Day-to-Day Workflow

### 10.1 Regular Development

1. Developers write conventional commits as usual (`feat:`, `fix:`, etc.)
2. CI validates syntax, linting, repo hygiene (existing pipeline)
3. PRs show changelog preview comment automatically

### 10.2 Creating a Release

**Option A — Local release:**
```bash
Rscript scripts/release.R --bump patch --name "Trait Provenance"
git push --follow-tags
```

**Option B — GitHub Actions:**
1. Go to Actions → Release → Run workflow
2. Select bump type (patch/minor/major)
3. Optionally set version override and release name
4. Workflow commits, tags, pushes, and creates GitHub Release

### 10.3 Preview Mode

```bash
Rscript scripts/release.R --bump patch --dry-run
```

Prints: new version, files that would change, changelog preview. No modifications.

### 10.4 Regenerating Docs Only

```bash
Rscript scripts/generate_changelog.R
Rscript scripts/generate_api_reference.R
```

Can be run independently without a version bump.

## 11. Constraints & Decisions

| Decision | Rationale |
|----------|-----------|
| Pure base R, zero dependencies | No need to install packages on CI or locally; scripts work anywhere R is installed |
| Custom Roxygen2 parser (not roxygen2 package) | EcoNeTool is a Shiny app, not an R package — roxygen2 requires DESCRIPTION file |
| Full CHANGELOG regeneration (not append) | User preference; ensures consistency; git history is the source of truth |
| Repo URL from `git remote` | Avoids hardcoding; works for forks |
| `app.R` updated by pattern match, not line number | The version block spans multiple lines and may shift as code evolves |
| Release script does NOT auto-push | Destructive remote action; user should explicitly `git push --follow-tags` |
| Git tags are annotated (not lightweight) | Annotated tags store tagger info, date, message; required for GitHub Releases |
